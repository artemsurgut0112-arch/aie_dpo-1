# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- Использован датасет `S12-hw-dataset.csv` с целевой колонкой `target` (почасовые данные).
- Организован `train / validation / test` по времени: 70% train, 15% val, 15% test.
- Сравнивались подходы: B1 (naive-last), B2 (moving-average), B3 (Ridge на признаках), R1 (GRU).
- Лучший подход — R1 (GRU) по метрике val_MAE.

## 2. Среда и воспроизводимость

- Python: 3.12
- pandas / numpy / scikit-learn: latest
- torch: 2.10.0
- Устройство (CPU/GPU): cpu (или cuda если доступна)
- Seed: 42
- Как запустить: открыть `HW12.ipynb` и выполнить Run All.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: почасовая
- Горизонт прогноза: 1 шаг вперед
- Размер датасета: 4320 строк
- Диапазон дат: 2025-01-01 – 2025-06-29
- Есть ли дополнительные внешние признаки: нет, только `target` и производные
- Комментарий: Ряд имеет тренд и сезонность, пропуски обработаны ffill/bfill, выбросы не удалялись, прогноз сложен из-за нестационарности.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Выполнен последовательный split: первые 70% по времени — train, следующие 15% — val, последние 15% — test.
- Доли: 70/15/15.
- Почему `random split` здесь некорректен: потому что нарушает временной порядок, позволяя модели "заглядывать" в будущее, что приводит к переобучению и нереалистичным оценкам.

### 4.2. Признаки для baseline-моделей

- Лаги: `lag_1`, `lag_7`, `lag_14`
- Rolling-признаки: `rolling_mean_7`, `rolling_std_7`
- Календарные признаки: `dayofweek`, `is_weekend`
- Обработка пропусков: dropna после построения признаков
- Масштабирование: StandardScaler, обучен только на train.

## 5. Модели и эксперименты (B1, B2, B3, R1)

- B1 (`naive-last`): прогноз равен последнему значению.
- B2 (`moving-average`): скользящее среднее по 7 шагам.
- B3 (`ridge-lag-features`): Ridge на лаговых, rolling и календарных признаках.
- R1 (`gru-forecast`): GRU на последовательностях длины 14.

Дополнительно:

- Основная метрика выбора: val_MAE
- Дополнительные метрики: RMSE, MAPE
- `window_size` для `GRU`: 14
- Batch size: 32
- Максимальное число эпох: 60
- Optimizer: Adam
- Learning rate: 0.001
- Сохранение лучшей модели: по минимальному val_loss.

## 7. Анализ

Наивный baseline B1 оказался сильным из-за простоты и отсутствия шума. Moving average B2 дал худшие результаты из-за сглаживания. Ridge на признаках улучшил качество, но GRU превзошла всех на val благодаря последовательному моделированию. Возможные утечки: в признаках rolling и lag использовались только прошлые данные, без будущего; в GRU окна строились без пересечения train/val/test.

## 8. Итоговый вывод

GRU — лучший для прогноза, но требует мониторинга. Рекомендуется использовать с roll-forward validation.
